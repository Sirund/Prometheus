The Problem

Indonesia sits on the Ring of Fire.
Most disaster apps don't.

270 million people face earthquakes, tsunamis, volcanic eruptions, and floods every year. The first thing that fails in a disaster is the network — and that's exactly when existing alert apps go silent.

📡

Alerts arrive too late

Standard push-notification systems require working mobile data. In a major earthquake, cell towers are the first infrastructure to fail. By the time an alert reaches you, you may already be in danger.

Network outages average <4 minutes after a major seismic event
🗺️

Guidance is generic

Generic "go to high ground" messages ignore the actual epicentre, magnitude, depth, and tsunami potential in the BMKG payload. People need specific, grounded instructions tailored to the actual event.

BMKG provides structured seismic data that no consumer app uses for AI briefings
♿

Accessible guidance is absent

In a disaster, reading a screen is hard. Visually impaired users have almost no options. High-stress situations demand eyes-free, hands-free, voice-first interfaces — which no existing disaster app provides.

Vision + speech modality fills a critical accessibility gap
🧠

AI is locked in the cloud

LLM-powered guidance requires a server call. Post-disaster connectivity is unreliable for hours or days. Any app that depends on an API endpoint is useless the moment you need it most.

LiteRT LM enables Gemma 4 fully on-device — no API key, no latency, no outage
The Solution

One app. Every layer of the crisis pipeline.

Prometheus doesn't rely on the cloud. It fuses BMKG open data with Gemma 4 running on the phone itself — so alarm, briefing, and evacuation guidance fire in under 30 seconds, offline.

📡

Poll BMKG feeds

Continuously reads autogempa.json and gempaterkini.json. Classifies events by magnitude, depth, felt intensity, and tsunami potential against configurable danger thresholds.

🚨

Trigger alarm + briefing

On a dangerous classification: audible alarm fires and Gemma 4 generates a spoken emergency briefing grounded in the actual BMKG payload — what happened, what to do, what to avoid.

🗺️

Plot evacuation route

Pins the epicentre on Google Maps, draws the hazard radius, and computes the shortest path from the user's location to outside the danger zone via the Directions API.

💬

Offline survival assistant

A fully offline chat powered by Gemma 4 with a survival system prompt — first aid, water, shelter, evacuation — available at any time, with or without internet.

Built on a single model that never leaves the device.

Gemma 4 (E2B, ~2.4 GB) runs via LiteRT LM with GPU acceleration. After a one-time download, it handles emergency briefing, survival chat, and camera-to-voice accessibility — all without a single outbound request.

Capabilities

Five screens. One mission.

Every feature is designed around a single constraint: it must work when the network is down and your hands are shaking.

📡

BMKG Live Monitor

Polls BMKG open JSON feeds and classifies events against configurable magnitude, depth, and tsunami-potential thresholds. The danger banner shifts from blue → orange → red as severity escalates.

autogempa.json · gempaterkini.json
🚨

Danger Alarm + Gemma 4 Spoken Briefing

When a dangerous event is classified, an audible alarm fires and Gemma 4 generates a spoken emergency briefing in under 30 seconds. The briefing is grounded in the real BMKG payload — magnitude, location, depth, felt intensity, tsunami potential — not a generic template. Delivered via AVSpeechSynthesizer, no screen required.

BMKG → Danger Rules → Gemma 4 Emergency Prompt → TTS
🗺️

Evacuation Routing

Plots the BMKG epicentre on Google Maps, overlays the configurable hazard radius, and computes the shortest route from the user's GPS location to outside the danger zone via the Directions API.

Google Maps SDK · Directions API · CoreLocation
💬

Offline Survival Chat

A streaming conversation with Gemma 4 using a survival-focused system prompt covering first aid, shelter, water sourcing, and Indonesia-specific hazard awareness. Works with zero connectivity after the initial model download.

Gemma 4 · LiteRT LM · Survival System Prompt
👁️

Vision Accessibility Mode

Camera frame → Gemma 4 vision → spoken response. Point the phone at surroundings, signage, or injuries. Gemma describes what it sees in calm plain language via TTS. No typing. No reading. Designed for visually impaired users and high-stress situations where a screen is impractical.

What Makes It Different

Three ideas that make Prometheus
a genuinely new thing.

01

Offline-first AI reasoning

Gemma 4 is downloaded once and never queried remotely. The LiteRT LM engine loads the model into GPU memory and keeps it there — so every alarm, every chat message, and every camera description runs in under a second, with zero network dependency. This is the only disaster app where the AI gets faster when connectivity disappears.

02

BMKG-grounded emergency briefings

Most AI assistants answer disaster questions generically. Prometheus injects the actual BMKG structured payload — coordinates, magnitude, depth, felt intensity, tsunami potential — directly into Gemma's emergency system prompt. The model reasons about the real event, producing guidance that is specific to the quake that just happened 12 km beneath you, not a template for "a large earthquake."

03

Eyes-free, hands-free survival interface

Prometheus treats voice as the primary output and camera as a primary input from the start, not as an afterthought. The Vision Accessibility mode — camera → Gemma 4 multimodal → spoken response — was designed from the ground up for visually impaired users and for anyone in a dark, smoky, or hands-occupied environment where reading a screen is simply not possible.
